### 1. GraphQL Introspectionスキーマの扱い方

- **苦労度**: ★★★☆☆（3/5）
- **深刻度**: ★★☆☆☆（2/5）
- **原因カテゴリ**: 外部依存
- **影響範囲**: Output System固有

#### 何が起きたか

GraphQL Introspection Query（`__schema`）で取得したスキーマ情報（Type、Field、Argument、Kind等）をLLMコンテキストに効率的に組み込む際、以下の問題が発生した。

1. Introspectionの返却JSONが非常に大きい（複雑なスキーマで数MB）
2. LLMのコンテキストウィンドウ圧縮を避けるため、スキーマ情報を圧縮・フィルタリングする必要があった
3. Fieldの nested type（List/NonNull wrapper）を正確に解析・表現しないと、LLMが型を誤解して不正なクエリを生成する

#### 原因

GraphQL仕様では、型情報が TypeKind（OBJECT, SCALAR, ENUM等）で分類され、FieldにはArgumentとしてInputTypeが含まれます。これらを「LLMが理解できる自然言語形式」に変換する非自明なマッピングが必要でした。

一般的なGraphQL ClientライブラリはIntrospecitonを返すだけで、人間向けのドキュメント生成ツール（GraphQL Playground等）しか参考になるものがなく、実装時に試行錯誤が必要でした。

#### どう解決したか

1. **スキーマ圧縮**: 必須でないメタ情報（__typename, description等）を削除
2. **種別別フィルタリング**: QUERY、MUTATION、SUBSCRIPTIONの3種に分けて、QUERY型のTypeと Fieldだけを抽出
3. **型情報の自然言語化**: `NonNull(List(String))` を `[String!]!` の形式で表記し、LLMへのプロンプトで「角括弧は配列、`!`は必須を意味する」と明記
4. **Argument情報の簡潔表記**: `field(arg1: String, arg2: Int): [Result!]!` の形式で1行で表現

実装箇所: `/workspace/target_repo/output_system/backend/src/db/schema-loader.ts` - `flattenGraphQLSchema()` 関数

#### 改善提案

なし。GraphQLの仕様を前提とした実装であり、別プロジェクトでも同様の処理が必要です。ただし、テンプレート化できる部分があります。

| 項目 | 内容 |

| 対象ファイル | `.claude/skills/develop-operations/references/graphql-schema-flattening-template.md` |
| 変更種別 | 新規追加 |

### 2. GraphQLクエリ結果のフラット化ロジック

- **苦労度**: ★★★★☆（4/5）
- **深刻度**: ★★★☆☆（3/5）
- **原因カテゴリ**: 実装困難
- **影響範囲**: Output System固有

#### 何が起きたか

GraphQL APIはレスポンスとしてネストされたJSON構造を返すことが多いが、テーブル・グラフ表示では 2次元フラットなデータ構造（行 × 列）が必要です。

実装時、以下のシナリオで問題が発生しました。

1. **ネストの深さが不定**: `user.profile.address.city` のように深さが API設計に依存
2. **配列ネスト**: `users[].orders[].items[].name` のような多層配列のデータをテーブル化する際、展開方法が複数あり、どの方針を採用すべか判断が必要
3. **スカラー値のみ抽出**: オブジェクト型フィールドはテーブルに表示できないため、スカラー値（String, Int, Float, Boolean）のみを自動抽出する必要があった

#### 原因

GraphQL仕様には「レスポンスのフラット化方法」が定義されておらず、アプリケーション次第です。

一般的なBI ツール（Tableau等）も同じ問題を抱えており、ツール側で「最大 N層展開」「配列は外側の 1層のみ展開」といったオプションを提供しています。

#### どう解決したか

1. **最上位の配列のみ展開**: GraphQLレスポンスの最上位フィールドが配列の場合、その配列を展開してテーブルの「行」とする
2. **スカラー値のみ抽出**: 各行に対して、スカラー型フィールド（String, Int, Float, Boolean）のみを抽出し、テーブルの「列」とする
3. **ネストは無視**: オブジェクト型フィールドはテーブル表示時にスキップ。グラフ表示時はスカラー値のみを数値化
4. **LLM指示**: プロンプトに「フラットなデータ構造（配列 + スカラー）を返すクエリを生成してください。オブジェクトネストは避けてください」と記載

実装箇所: `/workspace/target_repo/output_system/backend/src/db/query-executor.ts` - `flattenGraphQLResponse()` 関数

#### 改善提案

このアプローチは特定のGraphQL APIの設計に依存するため、テンプレート化は難しいですが、フロントエンド側でのフラット化ロジックを強化する改善案があります。

| 対象ファイル | `.claude/rules/constraints.md` |
| 変更種別 | 既存修正（末尾に追記） |

**具体的な変更内容:**

GraphQLレスポンスのフラット化処理で複数配列が存在する場合の対応方針を追記する:

> ## GraphQLレスポンスの複数配列フラット化
>
> GraphQL レスポンスに複数の配列フィールドが存在する場合、「Cartesian product」を避けるため、最上位の配列のみを展開すること。

>   }
> }

### 3. GraphQL接続テスト時の遅延と表示UX

- **苦労度**: ★★☆☆☆（2/5）
- **深刻度**: ★★☆☆☆（2/5）
- **原因カテゴリ**: ツール制約
- **影響範囲**: ルール改善で予防可能

#### 何が起きたか

DB管理モーダルで「接続テスト」ボタンをクリックした際、GraphQL の Introspection Query を送信するのに数秒要することがあり、その間ボタンの表示状態が不明確でした。

具体的には:
1. ボタンをクリックすると、ネットワークリクエストが送信される
2. 外部の GraphQL API の遅延（1-3秒）が発生
3. ユーザーからは「ボタンが効いたのか、それとも無応答か」が不明
4. ローディング表示がなく、ユーザーが複数回クリックしてしまう可能性

#### 原因

フロントエンドのボタン状態管理で「ローディング中」という中間状態を実装していなかったため、ボタン押下時から テスト完了時までの間、UI上に「処理中」という情報が表示されていませんでした。

これはツール制約というより、実装側の不備ですが、非同期操作（Fetch/Async）の長時間化に対するUXパターンが Rules に記載されていなかったため、実装時に見落とされました。

#### どう解決したか

フロントエンド（React）で以下の改善を施しました。

1. **ボタン状態管理**: `isLoading` state を追加
2. **ローディング表示**: ボタンテキストを「接続テスト中...」に変更、ボタン自体を disabled 状態に
3. **タイムアウト設定**: 10秒以上応答がない場合は自動的に「失敗」と判定し、ユーザーに通知
4. **エラーハンドリング**: NetworkError 等でも適切にメッセージ表示

実装箇所: `/workspace/target_repo/output_system/frontend/src/components/DbManagementModal.tsx` - `handleConnectionTest()` メソッド

#### 改善提案

非同期操作の UX パターンを Rules に追加することで、同様の問題を他のプロジェクトで予防可能。

| 変更種別 | 既存修正（末尾に追記） |

**具体的な変更内容:**

非同期操作のローディング状態表示・タイムアウト処理をRulesに追加する:

> ## 非同期操作のUXパターン
>
> フロントエンドで外部API呼び出しやネットワークリクエストが発生する場合、以下のUXパターンを必ず実装すること。

> ### ローディング状態表示（必須）

>    - スピナーアイコン表示（オプション）
>
> 2. **プログレスバー/スピナー表示**
>    - ローディングスピナーをモーダル or ボタン横に表示
>    - 「処理中...」テキストを併記
>

>
> 10秒以上応答がない場合は、自動的にタイムアウトエラーと判定し、ユーザーに通知すること。

## まとめ

v1.1 GraphQL対応の実装を通じて、以下の知見を得ました。

1. **GraphQL固有の複雑性**: Introspection スキーマの扱い、レスポンスのフラット化は、SQL ベースの実装には存在しなかった新たな課題。これらのアプローチはテンプレート化・ドキュメント化により、次の GraphQL 対応プロジェクトで活用可能。

2. **UX のための非同期操作パターン**: 外部 API の遅延を考慮した UI 設計パターンは、Web アプリケーション全般で応用可能。Rules に追加することで、他プロジェクトでの同じ問題を予防可能。

3. **テスト駆動による品質確認**: E2E テスト（Playwright）により、GraphQL Mutation 禁止、エラーハンドリング、接続テストといった要件を確実に検証でき、リグレッションを防止できました。
