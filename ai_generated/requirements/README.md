## 確定要件

### システム概要

- **システム名**: DataAgent
- **概要**: Databricks Genieのような自然言語データ分析システム。チャット形式で自然言語の質問を入力すると、SQLまたはGraphQLクエリを自動生成・実行し、結果をグラフやテーブルで可視化する
- **対象ユーザー**: ITリテラシーの高低に関わらず利用可能（マニュアルなしで使える説明的なUI）
- **配布形態**: 社内利用のみ
- **リリース段階**: まずは検証用（PoC）としてAPI連携機能を提供

### 機能要件（既存: DBモード）

1. **自然言語チャット入力**: チャット形式のUIで自然言語の質問を入力
2. **SQL自動生成**: Claude API（Anthropic）を使用して自然言語からSQLを自動生成
3. **SQL表示**: 生成されたSQLをユーザーに表示（透明性確保）
4. **SQL実行**: SELECTクエリのみ実行可能（読み取り専用）
5. **結果可視化**: クエリ結果を以下の形式で表示
   - 棒グラフ
   - 折れ線グラフ
   - 円グラフ
   - テーブル（一覧表示、データ幅に合わせた自動列幅調整）
6. **グラフ種類自動判定**: LLMがデータ特性から最適なグラフ種類を自動提案
7. **クエリ履歴**: 過去の質問と結果を保存・再利用可能
8. **ストリーミング応答**: ChatGPTのように応答が逐次表示される
9. **スキーマ自動取得**: DB接続時にINFORMATION_SCHEMAからテーブル・カラム情報（テーブルコメント・カラムコメント含む）を自動取得しLLMコンテキストに渡す
10. **エラーハンドリング**: 不正SQL生成時はエラー表示+再試行ガイドを提示
11. **AI分析コメント**: クエリ結果に対してLLMがデータの傾向・特徴・注目ポイントを自動分析し、コメントをストリーミング表示
12. **会話コンテキスト維持**: 同一会話内の過去のやり取り（生成SQL含む）をLLMに渡し、直前のSQLに対する修正依頼に対応

### 機能要件（追加: APIモード）

13. **OpenAPI→GraphQLゲートウェイ**: OpenAPI 3.0 Specファイルを読み取り、openapi-to-graphql（IBM製、MITライセンス）を使用してGraphQLスキーマを自動生成するゲートウェイを構築
14. **自然言語→GraphQLクエリ生成**: LLMがGraphQLスキーマを参照し、自然言語からGraphQLクエリを自動生成
15. **GraphQLクエリ実行**: 生成されたGraphQLクエリをゲートウェイ経由で対象REST APIに対して実行。GETリクエスト（読み取り専用）のみ許可
16. **GraphQLクエリ表示**: 生成されたGraphQLクエリをユーザーに表示（透明性確保、SQL表示と同様）
17. **APIデータ可視化**: API取得結果をDB結果と同じ形式（棒グラフ・折れ線・円グラフ・テーブル）で可視化。レスポンスのトップレベル配列のみをテーブル化
18. **データソース設定画面**: 設定画面でDB（SQL）とAPI（GraphQL）の切り替え、OpenAPI Spec登録（URL指定またはファイルアップロード）を行う
19. **複数API登録**: 複数のOpenAPI Specを登録し、会話ごとに使用するAPIを選択可能
20. **API会話コンテキスト維持**: DBモードと同様に、同一会話内の過去のGraphQLクエリをLLMに渡し、修正依頼に対応
21. **API会話履歴**: GraphQLクエリを履歴として保存（結果データは保存しない）
22. **APIエラーハンドリング**: API呼び出し失敗時はエラー表示+再試行ガイドを提示。タイムアウト・リトライはAI設計に委任

### 非機能要件

- **同時利用ユーザー数**: 1-10人（小規模）
- **対応ブラウザ**: Chrome のみ
- **対応デバイス**: PCのみ（レスポンシブ不要）
- **UIテーマ**: ライトモードのみ
- **言語**: 日本語のみ
- **認証**: なし
- **エクスポート**: 不要（画面表示のみ）
- **対象APIの認証**: なし（認証不要の公開APIが対象）
- **OpenAPI Specバージョン**: OpenAPI 3.0のみ対応

### 技術選定

| カテゴリ | 選定技術 | 選定理由 | 却下した代替案 |
|---------|---------|---------|---------------|
| フロントエンド | React + TypeScript | グラフ描画ライブラリが豊富、エコシステムが成熟 | Vue.js, Next.js |
| バックエンド | Node.js + Express + TypeScript | フロントエンドと言語統一、型安全 | Python + FastAPI, Go |
| LLM | Claude API (Anthropic) | SQL/GraphQL生成精度が高く、APIがシンプル | Gemini, GitHub Copilot |
| データベース | PostgreSQL / MySQL（両対応） | ユーザーの既存DB環境に合わせて選択可能 | 単一DB限定 |
| グラフライブラリ | Recharts | Reactネイティブで宣言的API、カスタマイズ容易、MITライセンス | Chart.js |
| OpenAPI→GraphQL変換 | openapi-to-graphql | IBM製、MITライセンス、実績あり | 自作実装 |
| デプロイ | Docker Compose | コンテナで一括起動、小規模に最適 | クラウドサービス |

### DB接続設定

- 環境変数で設定（.envファイル）
- 単一DB接続（複数DB切替不要）
- リードオンリー権限のDBユーザーで接続（Security要件）

### UIレイアウト

- サイドバー+メイン構成
  - 左サイドバー: クエリ履歴一覧
  - 右メイン: チャット画面 + グラフ/テーブル表示エリア
- 設定画面（新規追加）
  - データソース切り替え（DB / API）
  - OpenAPI Spec登録（URL指定 / ファイルアップロード）
  - 複数API管理（登録・選択・削除）

### 改修方針

- **既存DBモードの変更**: なし（既存機能はそのまま維持）
- **データソース分離**: 1つの会話では1つのデータソース（DBまたはAPI）のみ使用。混在利用は不可
- **APIモードのAI分析コメント**: 不要（DBモードのみ）
- **デプロイ構成**: 既存Docker Compose構成を維持。GraphQLゲートウェイはバックエンドプロセス内で動作
- **テスト方針**: モックAPIサーバーをDocker Composeに追加してテスト

### スコープ外（将来対応）

- ダークモード
- レスポンシブデザイン（スマホ対応）
- 結果のエクスポート（CSV/画像）
- 複数DB接続先の管理
- LLMプロバイダーの切り替え機能
- APIの認証方式対応（API Key、Bearer Token等）
- Swagger 2.0対応
- ネストされたJSONレスポンスの深層フラット化

## 専門家分析

### PO分析

- MVPスコープを明確に定義: 自然言語→SQL→グラフの基本フローに集中（既存）
- API連携は今回の改修スコープ: OpenAPI Spec → GraphQLゲートウェイ → 自然言語クエリ → 可視化
- 設定画面を新規追加してデータソース管理を実現
- 複数API登録・切り替えをサポート
- APIモードでの結果データ保存は不要（クエリのみ保存）
- リリースはまずPoC段階。認証付きAPI対応は将来対応

### Architect分析

- React + TypeScript / Node.js + Express の統一スタック（既存維持）
- openapi-to-graphql（IBM製、MITライセンス）でOpenAPI→GraphQL変換を実現
- GraphQLゲートウェイはバックエンドプロセス内で動作（別コンテナ不要）
- APIモードのデータフロー: 自然言語 → LLM（GraphQLスキーマ参照）→ GraphQLクエリ生成 → ゲートウェイ経由でREST API呼出 → レスポンスのトップレベル配列をテーブル/グラフ化
- GETリクエストのみ許可（Mutation/POST等は禁止）
- API呼出のタイムアウト・リトライはAI設計に委任（推奨: 30秒タイムアウト、最大3回リトライ）
- モックAPIサーバーをDocker Composeに追加してテスト環境を構築
- OSSライセンス: openapi-to-graphqlはMIT。社内利用のみのため問題なし

### QA分析

- テスト対象ブラウザ: Chrome のみ（既存維持）
- APIモードのテスト: モックAPIサーバーに対するE2Eテスト
- GraphQLクエリ生成の正確性テスト
- 設定画面のUIテスト（OpenAPI Spec登録、データソース切替）
- APIエラーハンドリング（タイムアウト、404、500等）のテスト
- 既存DBモードの回帰テスト（改修による影響がないこと）

### Security分析

- APIモードはGETリクエストのみ許可（読み取り専用）。GraphQL Mutationは禁止
- 対象APIは認証不要の公開APIが対象（認証情報の管理は不要）
- OpenAPI Specファイルのバリデーション: 悪意あるSpecの注入を防ぐ基本チェック
- CORS設定は既存と同様に適切に行う
- GraphQLクエリのバリデーション: Query（読み取り）のみ許可。Mutation/Subscriptionは拒否
